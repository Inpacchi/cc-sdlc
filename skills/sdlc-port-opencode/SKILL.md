# Port SDLC to OpenCode

Adapt an existing cc-sdlc installation so the project's SDLC skills, agents, and process documentation work with OpenCode's agent system. This creates a parallel `.opencode/` structure alongside the existing `.claude/` setup — it does not remove or replace anything.

**Argument:** `$ARGUMENTS` (optional: specific components to port, e.g., "agents only", "skills only")

## Preconditions

- The project must have an existing cc-sdlc installation (`.claude/skills/`, `.claude/agents/`, `CLAUDE.md` with SDLC sections)
- Verify with `ls .claude/skills/ .claude/agents/` and check `CLAUDE.md` exists
- If preconditions fail, tell the user to run `/sdlc-initialize` first

## Workflow

```
SURVEY → SCAFFOLD → AGENTS.MD → ADAPT AGENTS → ADAPT SKILLS → CONFIG → GAPS → REPORT
```

### 1. Survey Existing Installation

Read the project's current Claude Code SDLC setup:

1. `ls .claude/agents/` — list installed agent definitions
2. `ls .claude/skills/` — list installed skill directories
3. Read `CLAUDE.md` — identify SDLC sections (look for `## SDLC Process` header)
4. Read `.sdlc-manifest.json` if it exists — determine `[sdlc-root]` path
5. Check for existing `.opencode/` — if present, ask the user whether to overwrite or skip

Report the inventory to the user before proceeding.

### 2. Scaffold .opencode/ Structure

Create the OpenCode directory structure:

```
.opencode/
├── agents/        # Adapted agent definitions
├── skills/        # Adapted skill definitions
├── commands/      # Custom commands (empty initially)
└── tools/         # Tool configurations (empty initially)
```

Create each directory with `mkdir -p`. Add `.gitkeep` files to empty directories (`commands/`, `tools/`).

### 3. Generate AGENTS.md

Create `AGENTS.md` in the project root from the SDLC content in `CLAUDE.md`.

**What to copy:** All SDLC sections — process rules, verification policy, commit rules, debugging escalation, agent conventions. The goal is a complete OpenCode-compatible instructions file.

**Adaptation rules:**

| Pattern in CLAUDE.md | Replacement in AGENTS.md |
|---------------------|-------------------------|
| "CC (Claude Code)" | "OC (OpenCode)" |
| `.claude/agents/` | `.opencode/agents/` |
| `.claude/skills/` | `.opencode/skills/` |
| `.claude/agent-memory/` | (omit or note as unavailable) |
| `AskUserQuestion` tool references | "structured user prompt" (generic) |
| `Agent(subagent_type='...')` | "dispatch agent" (generic) |
| `SendMessage` tool | "inter-agent message" (generic) |
| Claude Code-specific tool table | OpenCode tool equivalents (see Tool Mapping) |

**Preserve unchanged:**
- All SDLC process rules and workflow documentation
- `[sdlc-root]` references (these resolve the same way for both tools)
- Verification policy, commit rules, debugging escalation
- Role definitions (CD stays "Claude Director" — the human role is tool-agnostic)

Add a header to AGENTS.md (substitute the actual resolved path from Step 1 for `{sdlc-root-path}`):

```markdown
<!-- Generated by sdlc-port-opencode from CLAUDE.md. Shared SDLC content lives in {sdlc-root-path}/. -->
```

**Note:** In the adapted body content, keep `[sdlc-root]` as a literal placeholder — it is resolved at runtime by the SDLC framework, the same way it works in CLAUDE.md. Only the header comment gets the concrete path for human orientation.

**Do NOT modify the original CLAUDE.md.**

### 4. Adapt Agent Definitions

For each `.md` file in `.claude/agents/` (skip `AGENT_TEMPLATE.md` and `AGENT_SUGGESTIONS.md`):

1. Read the agent file
2. Create a copy in `.opencode/agents/{name}.md`
3. Apply these adaptations:

**Frontmatter:**
- Claude Code: `description: "single line with \\n for newlines"`
- OpenCode: `description: >` folded scalar (multi-line, no escaping needed)
- Convert the description content, unescaping `\\n` to actual line breaks under the `>` scalar

**Tool references in the body:**

Apply the Tool Mapping table (§ Tool Mapping below). Replace specific Claude Code tool invocation patterns with generic action descriptions that OpenCode can interpret.

**Path references:**
- `.claude/agents/` → `.opencode/agents/`
- `.claude/skills/` → `.opencode/skills/`
- `.claude/agent-memory/{name}/` → note as unavailable or map to OpenCode equivalent if one exists

**Preserve unchanged:**
- All domain knowledge, rules, methodologies, and behavioral instructions
- `[sdlc-root]` references
- Knowledge routing phrases (these are part of the adapter contract)

### 5. Adapt Skill Definitions

For each skill directory in `.claude/skills/`:

1. Read `SKILL.md` from the skill directory
2. Create `.opencode/skills/{name}/SKILL.md` with adaptations
3. Copy any `references/` subdirectory contents unchanged (these are content, not tooling)

**Adaptations:**
- **Frontmatter:** Keep `>` folded scalar format (OpenCode uses the same convention) — no changes needed
- **Tool references:** Apply the Tool Mapping table
- **Agent dispatch patterns:** Replace `Agent(subagent_type='...')` with OpenCode agent invocation syntax
- **Skill invocation:** Replace `Skill` tool calls with OpenCode skill invocation
- **Inter-agent communication:** Replace `SendMessage` with OpenCode equivalent or flag as a gap
- **Path references:** `.claude/` → `.opencode/` where referring to the tooling layer

**Do NOT change:**
- `[sdlc-root]` references (shared content layer)
- Process logic, workflow steps, or domain rules
- Red Flags tables, Integration sections

### 6. Create opencode.json

Generate `opencode.json` in the project root:

```json
{
  "agents": {}
}
```

**Before writing:** Verify the current `opencode.json` schema URL and field names from OpenCode's documentation or `opencode --help`. Do not hard-code a schema URL without verification — OpenCode's conventions may differ across versions.

Populate the `agents` object with one entry per adapted agent in `.opencode/agents/`, mapping each to its file path and a default model.

If the project uses Context7 MCP (check CLAUDE.md for `context7` or `mcp__context7` references), add an `mcp` section wiring it.

### 7. Document Compatibility Gaps

Create `.opencode/COMPAT_GAPS.md` listing Claude Code features without direct OpenCode equivalents:

```markdown
# OpenCode Compatibility Gaps

Features from the Claude Code SDLC setup that may not have direct OpenCode equivalents.
Check your OpenCode version for current feature support.

| Claude Code Feature | Status | Workaround |
|--------------------|--------|------------|
| `Agent` tool (typed subagent dispatch) | Verify | Use OpenCode's native agent system |
| `SendMessage` (inter-agent messaging) | Verify | May need inline coordination |
| `AskUserQuestion` (structured prompts) | Verify | Use conversational questions |
| `EnterWorktree` / `ExitWorktree` | Verify | Manual `git worktree` commands |
| `TaskCreate` / `TaskUpdate` | Verify | External task tracking |
| `.claude/agent-memory/` (session state) | Likely absent | No persistent agent memory across sessions |
| `CronCreate` / `ScheduleWakeup` | Verify | External scheduling |
| `PushNotification` | Verify | Terminal notifications |
| `Monitor` (background streaming) | Verify | Bash polling loops |
```

Mark each as "Verify" — the user should check their specific OpenCode version. Where a feature is confirmed absent, note it as "Not available" with a concrete workaround.

### 8. Report

Output a summary:

```markdown
## OpenCode Port Summary

| Component | Source | Target | Count |
|-----------|--------|--------|-------|
| Agents | .claude/agents/ | .opencode/agents/ | N |
| Skills | .claude/skills/ | .opencode/skills/ | N |
| Instructions | CLAUDE.md | AGENTS.md | 1 |
| Configuration | — | opencode.json | 1 |
| Gaps doc | — | .opencode/COMPAT_GAPS.md | 1 |

### Manual Follow-up
- [ ] Verify OpenCode tool availability for items in COMPAT_GAPS.md
- [ ] Test a skill invocation in OpenCode to confirm wiring works
- [ ] Review AGENTS.md for any remaining Claude Code-specific language
```

## Tool Mapping

Use this table when adapting agent and skill files. Core file-operation tools share names; orchestration tools diverge.

| Claude Code | OpenCode | Adaptation |
|-------------|----------|------------|
| `Read` | `Read` | No change needed |
| `Write` | `Write` | No change needed |
| `Edit` | `Edit` | No change needed |
| `Bash` | `Bash` | No change needed |
| `Glob` | `Glob` | No change needed |
| `Grep` | `Grep` | No change needed |
| `LSP` | `LSP` | Verify availability; same concept |
| `Agent(subagent_type=...)` | OpenCode agent dispatch | Replace with generic "dispatch {agent}" language |
| `SendMessage` | (verify) | Replace with generic "message {agent}" or flag as gap |
| `Skill` tool | OpenCode skill invocation | Replace with generic "invoke {skill}" language |
| `AskUserQuestion` | (verify) | Replace with "ask the user" or structured prompt equivalent |
| `WebFetch` | `Fetch` or equivalent | Verify tool name in OpenCode |
| `WebSearch` | (verify) | Verify availability |
| `TaskCreate` / `TaskUpdate` | (verify) | May not exist — flag as gap |
| `EnterWorktree` / `ExitWorktree` | (verify) | May not exist — use manual git worktree |
| `Monitor` | (verify) | May not exist — use Bash polling |
| Context7 MCP | Context7 MCP | Same if MCP is configured in OpenCode |

**Adaptation strategy for "(verify)" tools:** Write the adapted file with generic action language ("dispatch the agent", "ask the user") rather than a specific tool name. This ensures the skill is functional regardless of which tools OpenCode actually supports. Add a comment `<!-- opencode: verify tool availability -->` next to adapted references so the user can update them once they confirm their OpenCode version's capabilities.

## Red Flags

| Thought | Reality |
|---------|---------|
| "I'll just copy the files without adapting tool references" | Unadapted Claude Code tool references will confuse OpenCode. Every file must go through the mapping. |
| "I'll remove the .claude/ directory to avoid confusion" | This skill creates PARALLEL compatibility. Never remove the Claude Code setup. |
| "The SDLC process content needs rewriting for OpenCode" | Process rules are tool-agnostic. Only tool interface references change. |
| "I'll modify CLAUDE.md to also work with OpenCode" | CLAUDE.md stays untouched. AGENTS.md is the parallel OpenCode instructions file. |
| "Agent domain knowledge needs adaptation" | Domain knowledge (rules, patterns, methodologies) is tool-agnostic. Only adapt the tooling interface. |
| "I'll skip the compatibility gaps document" | Users need to know what doesn't map cleanly. The gaps doc prevents trial-and-error frustration. |
| "OpenCode tools are identical to Claude Code tools" | File-operation tools share names. Orchestration tools (Agent, SendMessage, Skill) differ significantly. |
| "I should duplicate [sdlc-root] content for OpenCode" | `[sdlc-root]` content is shared. Both tools read from the same process docs, knowledge stores, and disciplines. Never duplicate. |
| "I'll skip the changelog update" | Changelog entries happen in the same step as the work. Not later. |
| "Frontmatter format doesn't matter" | Claude Code *agents* use double-quoted single-line descriptions with `\\n`; Claude Code *skills* use `>` folded scalars. OpenCode uses `>` folded scalars for both. Converting agent frontmatter format is required; skill frontmatter can stay as-is. |

## Integration

- **Depends on:** Existing cc-sdlc installation (`sdlc-initialize` must have run first)
- **Feeds into:** OpenCode-based development using the same SDLC process and knowledge layer
- **Uses:** `.claude/agents/`, `.claude/skills/`, `CLAUDE.md`, `.sdlc-manifest.json`, `[sdlc-root]` content
- **Complements:** `sdlc-initialize` (creates the Claude Code installation this skill reads from), `sdlc-migrate` (updates the Claude Code installation — re-run this skill after migration if OpenCode files need updating)
- **Does NOT replace:** `sdlc-initialize` (this adapts, doesn't install), `sdlc-migrate` (this is a one-time port, not ongoing sync)
- **DRY notes:** `[sdlc-root]` content (process docs, knowledge, disciplines, templates, playbooks) is shared between both tools and is NOT duplicated. Only the tooling interface layer (`.claude/` → `.opencode/`, `CLAUDE.md` → `AGENTS.md`) gets ported. After running `sdlc-migrate` to update the Claude Code layer, re-run this skill to sync OpenCode's tooling layer.
